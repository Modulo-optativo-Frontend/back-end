# ADR-003: Sistema de Autenticación JWT

## Contexto

La aplicación necesita un sistema de autenticación robusto para gestionar usuarios y proteger rutas. Se evaluaron diferentes estrategias:

- **Sesiones tradicionales** (cookie-based)
- **JWT (JSON Web Tokens)** con almacenamiento en cliente
- **OAuth 2.0** (Google, GitHub, etc.)

## Decisión

Usaremos **JWT (JSON Web Tokens)** con la siguiente implementación:

### Arquitectura de Tokens

| Token | Propósito | Duración | Almacenamiento |
|-------|-----------|----------|----------------|
| Access Token | Autorización de requests | 15 minutos | Memoria (no persistido) |
| Refresh Token | Renovación de sesión | 7 días | localStorage |

### Implementación

- **Biblioteca**: `jsonwebtoken` v9.0.2
- **Algoritmo**: HS256 (HMAC SHA-256)
- **Contraseñas**: `bcrypt` v5.1.1 con salt rounds configurables

### Flujo de Autenticación

```
1. Registro: POST /api/usuarios/registrar
   - Usuario envía name, email, password
   - Backend hashea password con bcrypt
   - Genera access + refresh tokens
   - Retorna tokens y datos del usuario

2. Login: POST /api/usuarios/login
   - Usuario envía email, password
   - Backend compara password hasheado
   - Genera tokens JWT
   - Retorna tokens y datos del usuario

3. Acceso protegido:
   - Cliente envía Access Token en header Authorization
   - Backend verifica firma y expiración
   - Si válido, permite acceso

4. Renovación: POST /api/usuarios/refresh
   - Cliente envía Refresh Token
   - Backend verifica y genera nuevo Access Token
   - No requiere re-autenticación completa
```

## Consecuencias

### Positivas

+ **Stateless**: No requiere almacenamiento de sesiones en servidor
+ **Escalable**: Los tokens se validan sin consulta a BD
+ **UX fluida**: Renovación transparente sin logout
+ **Compatible**: Funciona con apps móviles y SPA
+ **Seguridad**: Tokens de corta duración minimizan riesgo

### Negativas

- **Sin logout centralizado**: Refresh token válido hasta expiración
- **Almacenamiento vulnerable**: localStorage susceptible a XSS
- **Sin invalidación inmediata**: Cambios de contraseña no invalidan tokens activos

## Mitigaciones

- Access token corto (15 min) para limitar exposición
- Refresh token puede revocarse eliminando de BD
- Considerar HttpOnly cookies en futuro para mayor seguridad
