# ADR-004: Estructura de Componentes React

## Contexto

El frontend necesita una estructura de archivos organizada y escalable. Se evaluaron diferentes patrones:

- **Patrón atómico** (átomos, moléculas, organismos, templates, páginas)
- **Por funcionalidad** (features/modules)
- **Por tipo** (components, pages, hooks, utils) - estándar

## Decisión

Adoptar estructura **por tipo** con subdirectorios por dominio:

```
src/
├── components/
│   ├── cart/           # Componentes específicos del carrito
│   ├── home/           # Componentes de la página principal
│   ├── layout/         # Header, Footer, Layout
│   ├── product/        # Tarjetas, grids de productos
│   └── ui/             # Componentes UI genéricos (Button, Card)
├── pages/              # Páginas/rutas principales
├── lib/                # Utilidades y servicios (api, auth)
├── App.jsx             # Router principal
└── main.jsx            # Entry point
```

### Principios de Organización

1. **Componentes reutilizables en `/ui`**: Button, Card, Input, etc.
2. **Componentes de dominio en subcarpetas**: cart/, product/, home/
3. **Páginas en `/pages`**: Una carpeta por ruta
4. **Lógica en `/lib`**: API client, auth, helpers

### Naming Conventions

- **Componentes**: PascalCase (`ProductCard.jsx`)
- **Utilidades**: camelCase (`api.js`, `auth.js`)
- **Estilos**: Co-locados con componente o index.css global

## Consecuencias

### Positivas

+ **Curva de aprendizaje baja**: Estructura estándar de React
+ **Fácil navegación**: Localización intuitiva de archivos
+ **Escalable**: Nuevos componentes einfach hinzufügen
+ **Separación clara**: UI vs lógica vs páginas

### Negativas

- Puede crecer con muchos archivos en carpeta raíz
- Componentes relacionados pueden estar separados
